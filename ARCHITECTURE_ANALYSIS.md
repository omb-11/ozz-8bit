# Architecture Analysis

## Scope

This analysis covers the legacy MK1 repository contents found in:

- `MK1_CPU/`
- `bus_breakout/`
- `eeprom_programmer/`
- `helix_display_interface/`
- `start9_programming_interface/`

## Legacy MK1 Summary

The baseline project is a discrete-logic, EEPROM-microcoded 8-bit computer with companion Arduino-based interfaces and KiCad hardware.

## Findings

### CPU Architecture

- 8-bit shared data bus.
- 4 general-purpose registers: `A`, `B`, `C`, `D`.
- Separate `SP` and `PC`.
- Variable-length instructions: 1 or 2 bytes.
- Split address spaces with an overall 1 KB memory model.

### Bus Design

- Single common bus across CPU modules.
- LED-observable module state.
- External bus interface for up to two peripherals.
- Interrupt-capable peripheral signaling.

### Memory Model

- 1024-byte total addressable space.
- Code, stack, and data split into distinct regions.
- Code space is read-only in the later hardware revision.
- Direct and indirect addressing are both supported.

### Instruction Set

Observed from `MK1_CPU/programs/lib/mk1.cpu` and `MK1_CPU/code/microcode.py`.

- Register transfers with encoded source/destination fields.
- Immediate loads and stores.
- Basic ALU ops: `ADD`, `SUB`, `OR`, `AND`.
- Unary ops: `NOT`, shift, rotate.
- Conditional branches on `CF` and `ZF`.
- Stack operations through load/store special cases.
- Interrupt-related external read/write instructions (`exr`, `exw`) and jump-on-event instructions (`je0`, `je1`).

### ALU Design

- Two-operand ALU with dedicated operand register.
- Supports add, subtract, OR, AND, NOT, shift, rotate.
- Flag capture into a status register.
- No native multiply, divide, modulo, negate, or absolute value.

### EEPROM Microcode

- Control unit is generated by `MK1_CPU/code/microcode.py`.
- Up to 6 microsteps plus fetch/reset handling.
- Four EEPROMs store the 32-bit control word in byte lanes.
- Microcode is specialized per flag state for branch behavior.
- Interrupt behavior is encoded by conditionally rewriting special jump instructions.

### Display Interfaces

- Four-digit output register with EEPROM display decoder.
- `output_display.py` generates decimal and two's-complement display ROM contents.
- Separate Helix LCD display interface driven by ATmega328.

### Programming Interfaces

- Manual programming through switches.
- USB uploader via serial Arduino bridge.
- Start9 board stores multiple binaries in flash for offline loading.

### Upload Tools

- `uploader.py` streams `.bin` files over serial to Arduino-backed hardware.
- `flash_composer.py` packs multiple binaries into Start9 flash layout.

### Arduino Components

- CPU uploader sketch.
- Start9 programming interface sketch.
- Helix display interface sketch.
- EEPROM programmer sketch.

## Constraints in the Legacy Design

- The ISA is tightly coupled to MK1 control-line layout.
- Register selection and microcode encoding are constrained by the TTL implementation.
- Branching is limited to zero/carry conditions plus event-specific jump hooks.
- The memory model is too small for a modern educational software stack.
- Tooling is fragmented and centered on the legacy hardware flow.

## OZZ-8BIT Redesign Targets

- Replace the register model with `A`, `B`, `X`, `Y`, `SP`, `PC`, `FLAGS`, `TEMP`, `IR`.
- Move to a 64 KB unified address space with memory-mapped I/O.
- Introduce a new instruction encoding and modern educational ISA.
- Expand the ALU to include `MUL`, `DIV`, `MOD`, `NEG`, `ABS`.
- Replace ad hoc special instructions with explicit stack, branch, and interrupt instructions.
- Provide a clean assembler and emulator that are independent of MK1 source code.
- Define a modular hardware plan instead of keeping CPU logic embedded in one inherited board structure.
